## 第一章 模型总体概述

### 1.1 设计理念

本模型围绕 **Agentic Coding 的核心工作流**设计：Agent 需要**理解上下文** → **基于规约生成代码** → **在边界内安全执行** → **获得反馈循环** → **持续演进**。每个维度对应 Agent 工作链路中的一个关键环节。

**SDD（Specification-Driven Development，规约驱动开发）** 是本模型的核心范式。通过类型系统、接口契约和 Schema 校验构建机器可读的"规约护栏"，让 Agent 在严格定义的空间内自由发挥，实现"自由而不失控"的理想状态。

### 1.2 评估体系结构

评估模型采用 **"维度 → 子项 → 等级"** 三层结构：

- **维度层（5 个维度）**：定义评估的宏观视角，每个维度有独立权重
- **子项层（25 个指标）**：每个维度下 5 个具体评估指标，每项满分 10 分
- **等级层（5 个等级/项）**：每个子项有 5 个评分等级（卓越/优秀/良好/较差/缺失），附具体标准和证据要求

### 1.3 评分计算公式

```
维度百分制得分 = (该维度所有子项得分之和 ÷ 50) × 100

综合加权总分 = Σ (各维度百分制得分 × 该维度权重)

最终评级 = 根据综合加权总分查评级表
```

### 1.4 权重分配

| 编号 | 评估维度 | 权重 | 核心关注点 |
|:----:|---------|:----:|-----------|
| D1 | 🧠 上下文可理解性 | **30%** | Agent 能否快速理解项目全貌与代码意图 |
| D2 | 📐 规约驱动能力 (SDD) | **30%** | 是否有机器可读的规约定义行为边界（核心维度） |
| D3 | 🛡️ 边界控制与安全护栏 | **15%** | Agent 自主操作时是否有安全网防止越界 |
| D4 | ⚡ 任务可执行性 | **15%** | Agent 能否高效完成构建、运行、调试等任务 |
| D5 | 🌱 演进友好性 | **10%** | 是否支持 Agent 持续迭代和安全重构 |

### 1.5 综合评级标准

| 评级 | 分数区间 | 等级名称 | Agent 协作效能描述 |
|:----:|:-------:|:-------:|------------------|
| **S** | ≥ 90 | 卓越 | Agent 可近乎自主完成大部分任务，人工仅需最终审核 |
| **A** | 75–89 | 优秀 | Agent 高效协作，仅在复杂决策时需少量人工确认 |
| **B** | 60–74 | 良好 | Agent 可有效辅助，中等频率需人工介入修正方向 |
| **C** | 40–59 | 及格 | Agent 勉强可用，高频需人工纠偏，建议优先改进 |
| **D** | 20–39 | 较差 | Agent 难以有效工作，需先进行仓库基础设施建设 |
| **F** | < 20 | 极差 | Agent 基本无法使用，需从项目结构层面根本性改造 |

## 第二章 D1 — 🧠 上下文可理解性（Context Comprehensibility）

> **权重：30% ｜ 满分：50 分 ｜ 子项：5 个**
>
> 评估代码仓库是否提供了足够清晰、结构化的上下文信息，使 Agent 能够快速、准确地理解项目全貌、模块职责与代码意图。上下文是 Agent 一切工作的起点——如果 Agent 无法理解代码库，后续所有操作都将低效甚至有害。

### 1.1 项目全局描述文档

**评估目标**：仓库是否通过 README、ARCHITECTURE.md 等文档向 Agent 提供了项目目标、技术栈、核心模块关系、部署方式等全局性信息。Agent 需要在首次接触仓库时快速建立"心智模型"。

**评估要点**：

- [ ] 是否存在 README.md 且内容覆盖项目目标、技术栈、快速开始指南
- [ ] 是否有 ARCHITECTURE.md 或等效的系统架构说明文档
- [ ] 是否包含架构图（Mermaid / PlantUML / 图片形式的系统架构图、模块关系图）
- [ ] 项目核心术语/领域概念是否有 Glossary 或明确定义
- [ ] 文档是否与代码版本同步更新（非严重过时）

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|
| 9–10 | 🟢 卓越 | README 完整覆盖项目目标、技术栈、快速启动、模块关系；有独立 ARCHITECTURE.md 附架构图和数据流图；有 Glossary 定义核心领域术语；文档与代码版本同步 | 存在 ARCHITECTURE.md + 架构图 + 术语表 |
| 7–8 | 🔵 优秀 | README 清晰说明项目目标和主要模块；有基本架构说明但缺少架构图或数据流图；文档基本同步 | README 内容丰富，无独立架构图 |
| 4–6 | 🟡 良好 | README 存在但信息不完整，缺少模块关系或技术栈说明；无架构文档 | README 仅有安装步骤，缺少架构说明 |
| 1–3 | 🟠 较差 | 仅有项目名和一行描述，或 README 严重过时与代码不符 | README 最后更新超过1年或内容错误 |
| 0 | 🔴 缺失 | 无任何项目描述文件 | 无 README 或仅有空文件 |

### 1.2 目录结构清晰度

**评估目标**：评估目录组织是否语义化、层次分明，使 Agent 能通过目录结构快速定位代码位置和模块边界。良好的目录结构是 Agent 导航代码库的"地图"。

**评估要点**：

- [ ] 目录命名是否语义化（如 `src/services/` vs `src/s/`）
- [ ] 是否遵循领域驱动或功能驱动的分包策略
- [ ] 目录层次是否适中（不过深也不过浅，建议 3–5 层）
- [ ] 是否有目录说明文件（各目录下的 README 或根目录的 STRUCTURE.md）
- [ ] 测试目录与源码目录是否存在清晰的映射关系

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|
| 9–10 | 🟢 卓越 | 目录命名高度语义化，遵循领域/功能分包，层次 3–5 层合理；有目录说明文档或各目录 README；测试与源码目录映射清晰 | 有 STRUCTURE.md + 子目录 README |
| 7–8 | 🔵 优秀 | 目录结构合理可推断，命名语义化，但缺乏目录说明文档 | 目录自解释但无说明文档 |
| 4–6 | 🟡 良好 | 部分目录命名模糊（如 `utils/helpers` 过大）或层次扁平化 | 存在 `utils/` 超过 20 个文件的"垃圾桶"目录 |
| 1–3 | 🟠 较差 | 目录结构混乱，难以推断模块归属，命名随意 | `src` 下直接平铺数十个文件 |
| 0 | 🔴 缺失 | 所有文件平铺在根目录，无任何目录组织 | 根目录 30+ 文件无子目录 |

### 1.3 命名规范一致性

**评估目标**：评估文件名、类名、方法名、变量名是否遵循统一的命名约定，且名称具备自解释能力。一致的命名规范让 Agent 能够通过名称推断语义，减少对注释的依赖。

**评估要点**：

- [ ] 是否有明确的命名约定（ESLint 命名规则、EditorConfig、.clang-format）
- [ ] 文件命名风格是否统一（kebab-case / PascalCase / snake_case）
- [ ] 类/函数/变量命名是否自解释（无大量 `a`, `tmp`, `data`, `handler` 等通用名）
- [ ] 缩写使用是否一致且有据可查
- [ ] 命名是否反映业务域语言（Ubiquitous Language）

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|
| 9–10 | 🟢 卓越 | 有强制命名规则（ESLint/Ruff 配置），文件/类/方法/变量命名高度一致，名称自解释且反映领域语言，缩写有文档说明 | ESLint naming-convention 规则 + 无命名告警 |
| 7–8 | 🔵 优秀 | 大部分命名一致，核心模块命名自解释，偶有例外（如遗留代码） | 主干代码命名规范，个别旧模块风格不同 |
| 4–6 | 🟡 良好 | 命名风格混杂（如同时有 camelCase 和 snake_case）但主体可读 | 前后端风格不统一但各自内部基本一致 |
| 1–3 | 🟠 较差 | 大量缩写或无意义命名（如 `a`, `tmp`, `xxx`, `doStuff`） | 超过 20% 函数命名为 `handle/process/do` + 无语义后缀 |
| 0 | 🔴 缺失 | 命名完全随意，无任何规范，大量单字母变量 | 普遍使用 `a`, `b`, `x`, `temp` 等命名 |

### 1.4 内联文档与注释质量

**评估目标**：评估代码中注释和内联文档（JSDoc / docstring / XML Doc）的质量。注释应说明 **Why**（为什么这样做）而非 **What**（做了什么），关键函数应有参数/返回值文档。

**评估要点**：

- [ ] 公共 API 是否有 JSDoc / docstring / XML Doc 等标准文档注释
- [ ] 复杂算法或业务逻辑是否有 Why 注释（解释决策原因）
- [ ] 模块文件头部是否有用途说明
- [ ] 注释是否与代码同步更新（无过时/误导性注释）
- [ ] TODO / FIXME / HACK 标记是否规范使用且有追踪机制

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|
| 9–10 | 🟢 卓越 | 所有公共 API 有标准文档注释（含参数/返回值/异常说明），复杂逻辑有 Why 注释，模块头有用途说明，TODO 有 Issue 链接 | JSDoc 覆盖率 >90% + Why 注释 + `TODO(#issue)` |
| 7–8 | 🔵 优秀 | 公共 API 有文档注释，核心复杂逻辑有注释，但内部实现注释较少 | 公共方法有 docstring，私有方法部分缺失 |
| 4–6 | 🟡 良好 | 零散注释，质量参差不齐，部分模块有文档注释但不系统化 | 部分文件有注释，部分完全没有 |
| 1–3 | 🟠 较差 | 几乎无注释，或注释过时/误导 | 注释与代码逻辑不符或大量注释掉的代码块 |
| 0 | 🔴 缺失 | 完全没有注释 | 全仓库无任何注释行 |

### 1.5 依赖关系透明度

**评估目标**：评估项目内部模块间的依赖关系和外部依赖是否清晰、显式、可追踪。Agent 需要理解"改动 A 模块会影响哪些模块"才能安全地进行修改。

**评估要点**：

- [ ] 是否有依赖锁文件（package-lock.json / yarn.lock / poetry.lock）且版本明确
- [ ] 内部模块间依赖是否通过显式 import / DI 表达（无隐式全局变量耦合）
- [ ] 是否存在循环依赖并有检测机制
- [ ] 第三方依赖是否有选型说明或依赖审计记录
- [ ] 是否有依赖关系图或可通过工具自动生成

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|
| 9–10 | 🟢 卓越 | 依赖锁文件完整且 CI 中校验一致性；内部依赖通过 DI 或显式 import 清晰表达；有循环依赖检测（madge / deptrac）；有依赖选型文档 | lock 文件 + DI 容器 + 循环依赖检测 CI + 选型 ADR |
| 7–8 | 🔵 优秀 | 依赖锁文件存在且 import 路径清晰，无明显循环依赖 | lock 文件完整，import 结构清晰 |
| 4–6 | 🟡 良好 | 有依赖文件但存在少量隐式依赖或循环依赖 | 个别模块存在循环引用告警 |
| 1–3 | 🟠 较差 | 依赖关系混乱，全局变量或隐式耦合严重 | 大量 `window.xxx` 或隐式单例依赖 |
| 0 | 🔴 缺失 | 无依赖管理，随意引入外部代码，复制粘贴第三方代码 | 无 package.json / requirements.txt 或 vendor 未管理 |

## 第三章 D2 — 📐 规约驱动能力（Specification-Driven Development）

> **权重：30%（最高权重，核心维度）｜ 满分：50 分 ｜ 子项：5 个**
>
> SDD 范式强调通过类型系统、接口契约、Schema 校验等机器可读的"规约"来定义代码行为的合法边界。有了明确的规约，Agent 可以在规约空间内自由生成代码，而无需依赖模糊的自然语言指令——规约既是 Agent 的"导航地图"，也是其"行为围栏"。

### 2.1 类型系统完备性

**评估目标**：评估项目是否采用静态类型或类型注解系统，以及类型覆盖的深度和广度。类型系统是最高效的"规约"形式——Agent 可以直接从类型签名理解函数契约，并在编译阶段发现错误。

**评估要点**：

- [ ] 是否使用静态类型语言（TypeScript / Rust / Go / Java）或类型注解（Python typing / PHP 类型声明）
- [ ] 类型覆盖率是否达标（建议 strict 模式下 >95%）
- [ ] 是否使用高级类型特性（泛型、联合类型、条件类型、类型守卫）表达复杂约束
- [ ] `any` / `unknown` / `Object` 等逃逸类型的使用率
- [ ] 类型定义是否集中管理（如 `types/` 目录或 `.d.ts` 文件）

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|
| 9–10 | 🟢 卓越 | 全量 TypeScript strict / Pydantic BaseModel / Protobuf；类型覆盖率 >95%；善用泛型和高级类型；`any` 使用率 <1%；类型定义集中管理 | tsconfig `strict: true` + 类型覆盖率报告 + 极少 `any` |
| 7–8 | 🔵 优秀 | 核心模块有完整类型定义，覆盖率 >75%；有泛型使用但高级类型较少 | 主要模块有类型，个别 utility 用 `any` |
| 4–6 | 🟡 良好 | 部分类型定义，存在大量 `any`/`unknown`；或仅函数签名有类型，内部实现缺失 | 类型覆盖率 40–75%，`any` 使用较频繁 |
| 1–3 | 🟠 较差 | 少量类型注解，主要是动态类型；类型仅用于编辑器提示 | 仅入口文件有类型，内部全部 `any` |
| 0 | 🔴 缺失 | 纯动态语言且无任何类型提示 | 纯 JS 无 JSDoc `@type` 或纯 Python 无 typing |

### 2.2 接口契约定义

**评估目标**：评估对外 API 接口是否有机器可读的契约定义。接口契约是服务间通信的"法律文件"，Agent 可以基于契约自动生成客户端代码、Mock 数据和测试用例。

**例外**：如果当前评估项目不是以提供接口/API为功能，则此项目跳过，评分为满分。

**评估要点**：

- [ ] REST API 是否有 OpenAPI / Swagger 定义文件
- [ ] GraphQL 是否有完整 Schema 定义
- [ ] RPC 是否使用 Protobuf / Thrift 等 IDL
- [ ] 契约是否包含请求/响应示例（examples）
- [ ] 错误码是否枚举化并有文档说明
- [ ] 是否有 API 版本管理策略

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|
| 9–10 | 🟢 卓越 | OpenAPI 3.x / GraphQL Schema / Protobuf 定义完整；含请求/响应示例和错误码枚举；有 API 版本管理；契约可自动生成客户端 SDK | `openapi.yaml` 完整 + 示例 + 错误码枚举 + 版本化路由 |
| 7–8 | 🔵 优秀 | 有接口定义文件且覆盖主要 API，但示例或错误码不完整 | `swagger.json` 存在但缺少部分 examples |
| 4–6 | 🟡 良好 | 部分接口有文档（如 Postman Collection），但格式不统一或不完整 | 有 Postman 导出文件但非标准 OpenAPI |
| 1–3 | 🟠 较差 | 仅有零散 API 注释或 Wiki 页面描述 | API 信息散落在 README 和代码注释中 |
| 0 | 🔴 缺失 | 无任何接口定义或文档，API 行为仅能通过阅读源码理解 | 无 swagger / openapi / schema 文件 |

### 2.3 数据校验与 Schema 定义

**评估目标**：评估数据输入输出是否有 Schema 校验机制，且 Schema 是否与类型系统同源（Single Source of Truth）。数据校验是运行时的"规约执行器"，防止非法数据流入系统。

**评估要点**：

- [ ] 输入数据是否有校验（Zod / Joi / JSON Schema / Pydantic / class-validator）
- [ ] 数据库模型是否有 Schema 定义（Prisma / TypeORM / SQLAlchemy）
- [ ] 校验 Schema 与 TypeScript 类型是否同源（如 `z.infer` 导出类型）
- [ ] 校验错误信息是否友好且可被 Agent 解析
- [ ] 是否有 Schema 版本管理和迁移机制

**评分标准**：

| 分值 | 等级 | 评分标准 | 典型特征 / 证据 |
|:----:|:----:|---------|----------------|

## 第八章 改进优先级路线图

基于评估结果，按以下优先级顺序进行仓库改造，以最高效地提升 Agentic Coding 友好度。

### 8.1 P0 紧急改进（投入产出比最高）

**1. 创建 Agent 专属指引文件**：编写 CLAUDE.md / `.cursorrules`，明确项目约定、禁止操作、代码模板。这是成本最低、见效最快的改进。

**2. 补齐类型系统**：启用 TypeScript strict 模式或添加 Python typing，优先覆盖公共 API 和核心数据模型。

**3. 建立基本测试体系**：为核心业务逻辑编写单元测试，配置 CI 自动运行，确保 Agent 修改后可立即验证。

### 8.2 P1 重要改进

**4. 完善接口契约**：为 REST API 添加 OpenAPI 定义，为数据模型添加 Zod/Pydantic Schema。

**5. 配置 Lint 和格式化**：启用 ESLint/Prettier + pre-commit hook，统一代码风格。

**6. 优化目录结构**：按领域/功能分包，消除"垃圾桶"目录，添加目录说明。

**7. 完善构建自动化**：配置一键运行脚本（Makefile / docker-compose），确保 Agent 可自主构建和运行。

### 8.3 P2 持续优化

**8. 建立 ADR 机制**：开始记录架构决策，形成可追溯的决策链。

**9. 提升可观测性**：引入结构化日志和自定义错误类型体系。

**10. 模块边界强化**：引入 DI 容器，通过接口隔离模块，支持 Mock 替换。

以下是一个 Agent 专属指引文件的推荐模板，供评估后改造仓库时参考：

```markdown
# CLAUDE.md — Agent 工作指引

## 项目概述
- 项目名称：[名称]
- 技术栈：[语言、框架、数据库、部署平台]
- 架构模式：[分层架构 / 微服务 / Monorepo 等]

## 核心约定
- 代码风格：[遵循 ESLint + Prettier 配置，无需额外调整]
- 命名规范：[文件 kebab-case，类 PascalCase，变量/函数 camelCase]
- 分支策略：[GitHub Flow，feature/* 分支 → PR → main]
- Commit 规范：[Conventional Commits: feat/fix/docs/refactor/test]

## 禁止操作 ⛔
- 不要修改 `src/core/` 下的文件（核心模块，需人工审核）
- 不要直接操作数据库，必须通过 migration 文件
- 不要引入新的第三方依赖而不先讨论

## 常用命令
- `make dev` — 启动开发环境
- `make test` — 运行全部测试
- `make lint` — 运行 Lint 检查
- `make build` — 生产构建
- `make db-migrate` — 运行数据库迁移

## 新增功能模板
### 新增 API 路由
1. 在 `src/modules/[模块名]/` 下创建 controller / service / repository
2. 在 `src/modules/[模块名]/dto/` 下用 Zod 定义请求/响应 Schema
3. 在 `src/modules/[模块名]/__tests__/` 下编写测试
4. 在 `src/routes/` 中注册路由

## 测试约定
- 测试文件与源码同目录，命名 `*.test.ts`
- 使用 `describe('模块名')` + `it('should 行为描述')` 格式
- Mock 外部依赖，不 Mock 被测模块内部
```
